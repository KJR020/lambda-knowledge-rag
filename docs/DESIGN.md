# Knowledge RAG System 設計方針書

## 1. プロジェクト概要

### 1.1 目的

技術記事やドキュメントを収集し、RAG（Retrieval-Augmented Generation）システムに蓄積するLambda関数を構築する。
これにより、組織の知見を体系的に管理し、検索可能な形で活用できるようにする。

### 1.2 想定ユースケース

- 技術ブログ記事の自動収集と構造化
- 社内ドキュメントの知識ベース化
- マニュアル・仕様書の検索システム構築
- ナレッジマネジメントシステムの基盤

## 2. アーキテクチャ設計

### 2.1 全体構成

```
[S3 Upload] → [Lambda Function] → [Vector DB]
                    ↓
            [Embedding Service]
```

### 2.2 主要コンポーネント

#### 2.2.1 ドキュメント取り込み層

- **S3イベントトリガー**: ドキュメントアップロード時に自動起動
- **対応フォーマット**: PDF、Markdown、HTML、Plain Text、JSON
- **前処理**: ファイルサイズチェック、フォーマット検証

#### 2.2.2 ドキュメント処理層

- **テキスト抽出**: フォーマット別の抽出ロジック
- **メタデータ抽出**: タイトル、著者、作成日時、カテゴリ等
- **チャンク分割**: 意味的にまとまった単位での分割

#### 2.2.3 埋め込み生成層

- **プロバイダー選択**: AWS Bedrock / OpenAI
- **バッチ処理**: 効率的な埋め込み生成
- **次元数管理**: モデルに応じた次元数設定

#### 2.2.4 ベクトル保存層

- **ストア選択**: Pinecone / PostgreSQL+pgvector
- **インデックス管理**: 効率的な検索のための最適化
- **メタデータ保存**: 検索時のフィルタリング用

## 3. データフロー設計

### 3.1 処理フロー

1. **ドキュメント受信**
   - S3へのファイルアップロード
   - イベント通知によるLambda起動

2. **前処理**
   - ファイルサイズ検証
   - サポートフォーマット確認
   - ドキュメントID生成

3. **コンテンツ抽出**
   - フォーマット別処理
   - テキストクリーニング
   - 構造情報の保持

4. **チャンク生成**
   - 意味的な境界での分割
   - オーバーラップ設定
   - チャンクサイズ最適化

5. **埋め込み生成**
   - バッチ処理
   - エラーハンドリング
   - リトライロジック

6. **永続化**
   - ベクトルDB保存
   - メタデータ保存
   - 処理ログ記録

### 3.2 エラー処理フロー

- 各ステップでのエラーキャッチ
- DLQ（Dead Letter Queue）への送信
- リトライ可能/不可能エラーの分類

## 4. データモデル設計

### 4.1 ドキュメントモデル

```
Document {
  id: UUID
  source_key: S3キー
  source_bucket: S3バケット
  document_type: タイプ
  title: タイトル
  author: 著者
  content: コンテンツ
  metadata: メタデータ
  status: 処理ステータス
  created_at: 作成日時
  processed_at: 処理日時
}
```

### 4.2 チャンクモデル

```
Chunk {
  document_id: ドキュメントID
  chunk_id: チャンクID
  content: テキスト内容
  chunk_index: 順序
  embedding: ベクトル
  metadata: メタデータ
}
```

### 4.3 メタデータモデル

```
Metadata {
  file_info: ファイル情報
  content_info: コンテンツ情報
  classification: 分類情報
  source_info: ソース情報
  custom: カスタムフィールド
}
```

## 5. 技術選定の検討事項

### 5.1 埋め込みモデル選択

#### AWS Bedrock

**メリット:**
- AWSエコシステムとの統合
- 料金が使用量ベース
- 複数モデルの選択肢（Titan、Cohere等）

**デメリット:**
- リージョン制限
- モデルの更新頻度が低い

#### OpenAI

**メリット:**
- 最新モデルへのアクセス
- 高品質な埋め込み
- 豊富なドキュメント

**デメリット:**
- 外部API依存
- APIキー管理が必要

### 5.2 ベクトルDB選択

#### Pinecone
**メリット:**
- フルマネージドサービス
- スケーラビリティ
- 豊富な検索機能

**デメリット:**
- 外部サービス依存
- コスト（固定費用）

#### PostgreSQL + pgvector
**メリット:**
- オープンソース
- 既存DBとの統合可能
- コスト効率

**デメリット:**
- 自己管理が必要
- スケーリングの複雑さ

## 6. チャンク戦略

### 6.1 分割方針

- **意味的境界重視**: 段落、セクション単位
- **サイズ制約**: 1000文字程度（調整可能）
- **オーバーラップ**: 200文字（コンテキスト保持）

### 6.2 最適化考慮事項

- トークン数制限への対応
- 検索精度とのバランス
- 処理速度とのトレードオフ

## 7. 運用設計

### 7.1 監視項目

- 処理成功/失敗率
- 処理時間
- チャンク生成数
- API使用量
- エラー発生パターン

### 7.2 スケーラビリティ

- Lambda同時実行数制限
- バッチサイズの調整
- 非同期処理の活用
- SQSによるキューイング（大量処理時）

### 7.3 コスト最適化

- Lambda実行時間の最小化
- 埋め込みAPIコールの最適化
- S3ストレージクラスの選択
- ベクトルDB使用量の管理

## 8. セキュリティ考慮事項

### 8.1 データ保護

- S3暗号化
- 転送時の暗号化
- APIキーの安全な管理（Secrets Manager）

### 8.2 アクセス制御

- IAMロールによる最小権限
- VPCエンドポイントの活用
- 監査ログの有効化

## 9. 拡張性の考慮

### 9.1 将来的な機能拡張

- **マルチモーダル対応**: 画像、音声ファイルの処理
- **増分更新**: 差分処理による効率化
- **バージョン管理**: ドキュメントの更新履歴
- **検索API**: RAGシステムのクエリインターフェース

### 9.2 統合可能性

- **CI/CD連携**: ドキュメント更新の自動化
